# v1.2 Roadmap

**Preceded by**: v1.1 YAM multi-backbone pipeline (4 backbones trained,
evaluated, serving-verified on `yam_pick_up_grey_cube`).

## Inference optimization (model-level)

### torch.compile across all backbones
**Status**: PR #1 merged. FM 1.3×, DP 1.35×, SmolVLA 10×, X-VLA no effect.

### DiT/flow caching for X-VLA
**Status**: not started.
**What**: Reuse cached velocity predictions when cosine similarity between
consecutive ODE steps is high (DreamZero's key trick). Reduces effective
denoising steps from 10 to ~3-4 without retraining.
**Expected impact**: 2-4× on X-VLA (78ms → ~25ms).
**Effort**: medium — need to modify flow matching inference loop in
`modeling_xvla.py` to track and compare velocity norms.

### FP8/INT8 quantization
**Status**: not started.
**What**: Post-training quantization via bitsandbytes or torch.ao for
transformer-heavy VLAs (X-VLA, SmolVLA).
**Expected impact**: 1.5-2× memory reduction + 1.3-2× speedup.
**Effort**: medium — need calibration dataset, verify action accuracy
doesn't degrade, handle mixed-precision edge cases.

## Training quality

### SmolVLA underfitting on YAM
**Status**: diagnosed. SmolVLA replay MSE is 10× worse than X-VLA on both
arm joints AND grippers. The grip/arm error ratio (75×) is actually the
same as X-VLA — this is overall underfitting, not a gripper-specific issue.
**Possible causes**: 20k steps insufficient for 100M learnable params on
10 episodes; LR 1e-5 may be too conservative; batch 32 too small.
**Next steps**: hyperparameter sweep (40k steps, LR 2e-5, batch 64).
Low priority — FM already dominates on both accuracy and speed.

### Gripper MSE across all backbones
**Status**: known limitation. All 4 backbones show grip/arm MSE ratios
from 75× (VLAs) to 8879× (DP). Root cause: MSE regression on bimodal
open/close actions (0 or 2.4) averages to ~1.2 — never predicts the
actual value.
**Fix options**:
1. Binary classification loss for gripper dims (BCE instead of MSE)
2. Weighted per-dim loss (upweight gripper dims)
3. Discretize gripper to {0, 1} in dataset preprocessing

Options 1-2 require patching the policy's loss function in lerobot.
Option 3 is a dataset-level change. All are medium effort.
**Priority**: low for now — on-robot eval will reveal if this actually
causes grasp failures. The gripper state-action correlation is 0.98,
so the policy may still work despite high MSE.

## Deployment

### On-robot evaluation
**Status**: blocked on robot access.
**Protocol**: `lobe-serve` → limb's `WebSocketPolicyClient` → N trials
per checkpoint. Manual, batched when a backbone reaches Phase 3 pass.
**Priority**: high — this is the only evaluation that measures real task
completion. Replay MSE is a proxy.

### Multi-task YAM datasets
**Status**: user collecting new tasks.
**Pipeline**: validate → convert to image → `train_yam.py` → replay eval
→ serving test. The v1.1 pipeline handles this out of the box (just
change `--dataset.repo_id`).

## Non-goals for v1.2

- TensorRT/ONNX export (high effort, no standard VLA pipeline exists)
- Multi-task training across YAM datasets (wait for >1 dataset first)
- Sim environment for YAM (replay MSE + on-robot is the protocol)
